# point-machine — Архитектура

Бессерверный peer-to-peer синхронизатор файлов (альтернатива Syncthing). Файлы передаются
напрямую между устройствами одного пользователя. **Никакого нашего сервера нигде нет** — ни
для хранения, ни даже для обнаружения. Байты файлов никогда не проходят через третьи стороны.

Платформы: Android, iOS, Linux, macOS, Windows. Web сознательно не поддерживается (браузеры не
умеют UDP/DHT и настоящую фоновую работу).

---

## 1. Базовые принципы

- **Никакого сервера, никогда.** Обнаружение пиров заменено тремя бессерверными путями (LAN,
  WAN DHT, QR). Единственная внешняя инфраструктура — bootstrap-узлы DHT и опциональный STUN —
  публична, не хранит данных, заменяема и **отключаема** (тогда остаются только LAN + ручной QR).
- **Сквозное шифрование (E2E).** Каждый блок файла запечатывается ключом папки до того, как
  покинет устройство, и хранится зашифрованным на диске.
- **Соединяются только спаренные устройства.** Идентичность — пара ключей Ed25519; принимаются
  только явно спаренные Device ID.
- **Чистый Dart.** Никакого C/C++ в нашем коде. Flutter + Riverpod, `dart:io` + `sembast`.

---

## 2. Слои (`lib/`)

```
core/        идентичность, пейринг, конфиг, модели, пути, кодеки папок/шар
crypto/      AEAD (XChaCha20-Poly1305), вывод ключа папки, hex-кодек
storage/     FileStore — чтение/запись реальных файлов на диске
sync/        движок синка: индекс, сканер, блоки, версионные векторы, SyncService, SyncHost
transport/   обнаружение (mDNS, DHT), сигналинг, каналы TCP / WebRTC / Bluetooth, координатор
platform/    фоновое выполнение, трей, абстракция SyncController + мост между изолятами
state/        провайдеры Riverpod (связка UI ↔ синк)
ui/          экраны и виджеты (Material 3 Expressive)
i18n/        переводы slang
```

## 3. Идентичность и пейринг

У каждого устройства две пары ключей (`core/identity.dart`):

- **Ed25519**, ключ подписи → его идентичность. **Device ID** — это хеш публичного ключа
  подписи, поэтому он самоудостоверяющий.
- **X25519**, ключ согласования → для вывода общих секретов шифрования папок.

Ключи лежат в `identity.json`. **`PairingPayload`** (`core/pairing.dart`) — публичная часть,
которой устройство делится: `{deviceId, name, signingKey, agreementKey}`.

**Пейринг** устанавливает взаимное доверие обменом пейлоадами:

- **Вживую / LAN** — подключение к сигнальному порту пира и обмен `PairRequest` /
  `PairResponse`; короткий числовой **код пейринга** (`transport/pairing_code.dart`) позволяет
  обеим сторонам убедиться, что это один и тот же пир, по внешнему каналу.
- **Удалённо** — обе стороны анонсируются под infohash, выведенным из общего кода, находят друг
  друга и далее пейрятся так же.
- **QR** — пейлоад кодируется в QR для сканирования камерой.

Спаренные пиры сохраняются в `peers.json`.

## 4. Папки и swarm-секреты

У синхронизируемой папки (`FolderConfig`) есть случайный 32-байтный **swarm-секрет**
(`transport/swarm.dart`). Из него выводятся две вещи:

- **Infohash** = `sha256(secret)[0..20]` — ключ точки встречи для поиска пиров (имя LAN-сервиса
  и ключ анонса в DHT). По infohash о содержимом узнать ничего нельзя.
- **Ключ папки** — см. §7.

Шаринг папки отправляет `FolderShare` (`{folderId, label, swarmSecret}`) уже спаренному пиру,
который выбирает локальный путь для приёма. Права по пиру (`canSend` / `canReceive`) хранятся в
папке.

---

## 5. Обнаружение (поиск пиров)

Три пути дают один и тот же результат — `адрес:порт` пира, после чего открывается обычное
соединение.

| Путь | Механизм | Файл |
|------|----------|------|
| **LAN** | mDNS-мультикаст + UDP-broadcast маяк, анонсирующий `PairingPayload`, сигнальный порт и TCP-порт синка | `transport/lan_beacon.dart` |
| **WAN** | BitTorrent Mainline DHT: анонс/поиск под infohash папки, чтобы узнать IP:port пира | `transport/dht.dart` |
| **Пейринг** | код пейринга через QR / DHT | `transport/pairing_code.dart` |

> **DHT — это обнаружение, а не транспорт.** Она лишь сопоставляет `infohash → пиры` и заменяет
> центральный сигнальный сервер. Байты файлов через неё не идут.

---

## 6. Транспорт (передача байтов)

После того как пир найден, сначала открывается **сигнальный** канал
(`transport/lan_signaling.dart` — сокет с построчным JSON на порту **49322**). По нему
устройства обмениваются `SignalHello {infohash, deviceId}`, а для WebRTC — SDP/ICE.

Затем **`SyncTransportCoordinator`** (`transport/sync_transport.dart`) открывает сам `PeerLink`,
перебирая кандидатов по приоритету и используя первый, который соединился:

| Приоритет | Тип | Канал |
|----------:|-----|-------|
| 5  | **Прямой TCP** (LAN, хост-в-хост) | `transport/tcp_transport.dart` |
| 10 | **WebRTC data channel** (DTLS 1.3, NAT hole-punching, опциональный STUN) | `transport/negotiator.dart` |
| 20 | **Bluetooth** (офлайн-резерв) | `transport/bluetooth_transport.dart` |

Любой канал — это `PeerLink`: кадрированный упорядоченный поток сообщений `SyncMessage`.

## 7. Модель безопасности

| Слой | Защита |
|------|--------|
| **Идентичность** | ключи Ed25519; принимаются только спаренные Device ID |
| **В транзите** | TCP/WebRTC поверх DTLS 1.3 |
| **Сквозное (E2E)** | каждый блок запечатан **XChaCha20-Poly1305** ключом папки |
| **На диске** | блоки хранятся зашифрованными |

**Ключ папки** (`crypto/folder_key.dart`) выводится для каждой пары устройств:

```
shared    = X25519(мой_agreement_priv, agreement_pub_пира)
folderKey = HKDF-SHA256(shared, nonce = swarmSecret, info = "folder-key/v1", 32 байта)
```

Поскольку общий секрет X25519 симметричен, оба устройства выводят один и тот же ключ. Блоки
запечатываются/открываются `FolderCipher` (`crypto/aead.dart`). Точки STUN/TURN настраиваются
пользователем в настройках и отключаемы.

## 8. Движок синхронизации

Движок (`sync/engine.dart`) — компактный протокол поблочной репликации поверх `PeerLink`.

**Индекс.** У каждой папки есть индекс в `sembast` (`sync/index.dart`): на каждый файл —
`FileMeta` (`path, size, modified, blockHashes, deleted`) плюс **версионный вектор**. **Сканер**
(`sync/scanner.dart`) обходит папку, режет файлы на **блоки по 128 КиБ**, считает SHA-256 каждого
(в отдельном изоляте, `sync/blocks.dart`) и обновляет индекс.

**Определение конфликтов** — через **версионные векторы** (`sync/version_vector.dart`): счётчики
по устройствам с операциями `dominates` / `concurrentWith`. Параллельные правки одного файла
сохраняются как **конфликтная копия**, а не перезаписываются.

**Протокол обмена** (`transport/messages.dart`):

```
A ──IndexSnapshot──▶ B      обе стороны анонсируют свой индекс
B решает, какие записи новее, и для каждого недостающего блока:
B ──WantBlock(hash)─▶ A
A ──BlockPayload────▶ B      запечатанные байты блока
… B собирает файл, когда все блоки получены, и обновляет индекс
```

Хеширование блоков означает, что **неизменённые блоки не пересылаются** — едет только дельта.

## 9. Фоновое выполнение

Синк должен работать, даже когда окно приложения закрыто. На каждой платформе это решено
по-своему; UI зависит от единой абстракции **`SyncController`** (`platform/sync_controller.dart`)
с двумя реализациями:

```
              ┌──────────────── Android ────────────────┐   ┌──── Desktop ────┐
изолят UI  │  RemoteSyncController ──invoke/on──┐        │   │ LocalSyncController
           │                                    ▼        │   │        │
фон. изолят│            flutter_background_service       │   │        ▼
           │            SyncHost → SyncService           │   │ SyncHost → SyncService
           └─────────────────────────────────────────────┘   └────────┘
                 (foreground service, владеет синком)         (главный изолят, жив
                                                               за счёт трея)
```

- **Android.** Фоновый изолят `flutter_background_service` — **единственный владелец**
  `SyncService` и индексной БД. Два изолята не могут держать его одновременно: подерутся за
  порты mDNS/TCP/сигналинга, а два хендла `sembast` на один файл повредят БД. UI общается с
  фоновым изолятом через небольшой RPC-мост (`platform/sync_background.dart`,
  `remote_sync_controller.dart`, `sync_messages.dart`, `sync_codec.dart`): события, ближние
  пиры, счётчики файлов и интерактивные запросы pair/share пересекают границу изолятов.
- **Desktop.** OS-фонового изолята нет, поэтому `SyncService` работает в главном изоляте через
  `LocalSyncController`. Закрытие окна **сворачивает в трей**, а не завершает приложение
  (`platform/desktop_lifecycle.dart` + `desktop_tray.dart`), плюс регистрируется
  **автозапуск при входе**. Реальный выход — только из меню трея.
- **iOS.** Отложено — непрерывного фона нет, возможны лишь периодические проходы
  `BGTaskScheduler`.

`SyncHost` (`sync/sync_host.dart`) — Riverpod-независимый сборщик, который загружает
идентичность/конфиг/пиров/папки/БД с диска и запускает `SyncService`. Он общий для обеих
реализаций, поэтому один и тот же движок крутится и в фоновом изоляте (Android), и в главном
(desktop).

## 10. Управление состоянием

Riverpod (`state/`) связывает движок с UI. Ключевые провайдеры:

- `syncControllerProvider` — создаёт платформенный контроллер, подключает **исходящие** потоки
  (события, изменения папок, спаренные пиры, входящие запросы) к нотифаерам.
- `syncBindingProvider` — подключает **входящие** изменения (папки, пиры, конфиг, окно
  расписания) и шлёт контроллеру `reload*`. Вынесен из контроллера, чтобы не было циклической
  зависимости провайдеров.
- `foldersProvider`, `pairedPeersProvider`, `configProvider`, `syncEventsProvider`,
  `nearbyDevicesProvider`, `syncActiveProvider` (окно расписания).

Всё постоянное состояние — это обычные файлы в каталоге данных приложения (`identity.json`,
`peers.json`, `folders.json`, `point-machine.db`) плюс `SharedPreferences` для конфига — именно
поэтому headless фоновый изолят может пересобрать всё с диска.

## 11. Сквозной поток (пример для LAN)

```
1. Оба устройства анонсируются по mDNS (PairingPayload + порты).
2. Устройство A видит знакомого пира → открывает сигналинг (порт 49322) → SignalHello(infohash).
3. Координатор открывает канал: в LAN побеждает прямой TCP.
4. Обе стороны шлют IndexSnapshot. У A файл новее.
5. B запрашивает недостающие блоки по хешу; A потоком шлёт запечатанные BlockPayload.
6. B открывает каждый блок ключом папки, собирает файл, обновляет индекс.
7. SyncEvents уходят в UI (экран Activity); на desktop трей удерживает всё это после закрытия
   окна.
```
